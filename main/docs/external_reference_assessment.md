# 外部参考项目评估

评估日期：2026-05-12

本文件记录我们对三个外部项目的判断。结论是：可以借鉴架构与接口思路，但当前阶段不直接复制或内嵌外部项目代码。这样可以降低许可证、依赖体积、运行环境和后续维护风险。

## 总体结论

| 项目 | 是否借鉴 | 主要用途 | 是否直接引入代码 |
| --- | --- | --- | --- |
| OpenBMB/VoxCPM | 主路线，后续单独验证 | 真实 TTS、声音设计、声音克隆、微调路线 | 暂不直接引入 |
| CommentOut64/neo-tts | 仅保留架构参考 | 分段 TTS 工作流、音色管理、时间线播放 | 不引入运行后端 |
| entropy622/LLM_Live2D | 强借鉴 | LLM 输出结构化 JSON 控制 Live2D 表情与参数 | 暂不直接引入 |
| soiwm/astrbot_plugin_vtuber | 借鉴 | WebSocket 驱动渲染端、TTS 播放、音量驱动口型同步 | 暂不直接引入 |

## VoxCPM

仓库：https://github.com/OpenBMB/VoxCPM

### 适合借鉴的点

- 它是面向真实语音合成的 TTS 系统，目标明显比我们现在测试页里的浏览器合成音更接近人声。
- VoxCPM2 支持多语言、声音设计、可控声音克隆、48kHz 输出，适合作为比赛项目中“声音模块”的主路线。
- 项目提供 Python API、CLI、Web Demo、流式生成和微调入口，适合我们后续做独立语音服务验证。
- 它支持 SFT 和 LoRA 微调，官方说明少量音频也可以做说话人、语言或领域适配。对我们“自己训练/微调得到模型”的路线有参考价值。
- 许可证是 Apache-2.0，工程使用上相对友好，但仍要保留来源说明，不能把模型权重混进本仓库。

### 当前不适合直接塞进主工程的原因

- VoxCPM 的重型 GPU 依赖不适合直接并入当前 Python 3.11 主环境；板端也不具备 NVIDIA CUDA 运行条件。
- RK3588/Firefly 不具备 NVIDIA CUDA 环境，直接在板端跑完整 VoxCPM2 的概率很低。
- 模型体积、显存/内存占用、实时性都需要单独实验，不应该污染现在稳定的板端主环境。
- 声音克隆存在误用风险，项目中必须只使用授权声音素材，并在作品文档里说明合成音来源。

### 我们的落地策略

短期不再用“电子管”式浏览器纯音作为最终声音，只保留它作为嘴型调试占位。真实声音模块收敛到 VoxCPM2 路线。推进方式如下：

1. 定义统一 TTS 接口：输入文本、说话风格、情绪，输出音频文件或音频流，以及可选的口型/音量时间线。
2. 保留公网验证、项目内推理和 Gradio 兼容三条明确路径，避免把项目内模式误接到外部服务。
3. 单独建立可复现的 VoxCPM 实验环境，与轻量的 Python 3.11 板端运行环境隔离依赖。
4. 先在 Windows 或 GPU 机器上跑通 VoxCPM2 生成样例，再评估 RK3588 可行性，包括 ONNX、C++、量化或局域网自托管服务。
5. 比赛演示阶段如果 RK3588 无法实时跑完整 TTS，可采用“板端控制 + 本地预生成/局域网自托管 VoxCPM2 推理”的折中方案，但文档中必须说明这是自建模型服务，不是公网 API。

当前已把少量 VoxCPM2 保存音色样本和对应转写文本纳入 `assets/tts/voxcpm_samples/`。这些文本只作为 VoxCPM 参考音频的 `prompt_text`，不能替代正式训练数据。

## Neo TTS

仓库：https://github.com/CommentOut64/neo-tts

### 适合借鉴的点

- 它的段级可编辑语音合成工作流比浏览器 `speechSynthesis` 更接近真实声音生产工具。
- 它把长文本拆成独立语音段，支持插入、追加、修改、删除、重排和局部重推理，适合后续“机器人连续对话但只重生成局部回复”的工作流。
- 它强调段间停顿、边界融合和完整时间线，这对 Live2D 口型同步很有价值：前端不应该靠猜测字符节奏，而应该消费 TTS 后端给出的音频和时间线。
- 它支持同一篇文档混合多种音色，以及按段覆盖语速、top_k、top_p、temperature、noise_scale 等参数，适合我们后续做“情绪影响说话方式”的控制接口。
- 许可证是 Apache-2.0，可以作为架构参考；正式使用仍要核对底模、音色权重和参考音频的授权。

### 当前不适合直接塞进主工程的原因

- Neo TTS 是依赖 PyTorch/CUDA、Node.js 和 uv 的工作站式应用；其依赖范围不适合并入轻量的 Python 3.11 板端环境。
- 它是完整工作站，包含后端、前端、桌面启动器、模型管理和长文本编辑器。直接引入会明显扩大项目体积和维护面。
- RK3588 没有 NVIDIA CUDA，重型神经 TTS 推理很难直接放在 Firefly 上实时运行。

### 我们的落地策略

Neo TTS 只保留为“段级编辑、时间线和音色管理”的架构参考，不作为当前运行后端：

1. 前端只认 `POST /tts`：输入文本、语速、音色，返回音频。
2. 当前开发期由本地适配服务调用 VoxCPM Gradio 队列接口。
3. 当 TTS 后端能输出段级时间线时，口型从“文本估算”升级为“真实音频/时间线驱动”。

## LLM_Live2D

仓库：https://github.com/entropy622/LLM_Live2D

### 适合借鉴的点

- 它把 Live2D 接入 LLM 对话流程，让 LLM 在生成回复文本时同步输出表情和参数控制。
- 它不是让 LLM 随意写全部 Live2D 参数，而是要求 LLM 返回固定 JSON，再经过表情发现、参数白名单和范围裁剪。
- 它支持表情混合、参数覆盖、鼠标注视跟随、拖拽移动、缩放等展示特性，这些都符合我们的虚拟陪伴机器人展示需求。
- 它的核心思想是“LLM 只提出意图，本地控制层负责安全过滤和平滑执行”，这点非常适合比赛项目。
- 许可证是 MIT，适合参考实现方式。

### 当前不适合直接复制的原因

- 它是 React 19、TypeScript、Vite、PixiJS、pixi-live2d-display 技术栈；我们的板端主控当前是 Python 结构。
- 它偏前端网页实验项目，不包含我们需要的摄像头、麦克风、ASR、TTS、本地模型和 RK3588 部署链路。
- 直接复制会让项目变成两个不清晰的主工程，后续难维护。

### 我们的落地策略

我们要借鉴它的控制协议，而不是照搬 UI：

1. 设计 `RobotReply` 结构：包含 `text`、`emotion`、`expression_mix`、`motion`、`parameter_overrides`、`speech_style`。
2. 所有 LLM 输出必须先过 JSON Schema 校验。
3. Live2D 参数必须走白名单和范围裁剪，禁止 LLM 直接写任意参数。
4. 表情、动作和参数的最终值由本地控制器合成，并带平滑过渡。
5. 后续如果选择 Web 渲染端，可以用 Python 后端通过 WebSocket 控制浏览器/PixiJS 渲染窗口。

## astrbot_plugin_vtuber

仓库：https://github.com/soiwm/astrbot_plugin_vtuber

### 适合借鉴的点

- 它采用 Python 后端加 Electron 桌面客户端，通过 WebSocket 连接后端与 Live2D 渲染端。
- 功能链路包含消息监听、情感分析、TTS 集成、WebSocket、Live2D 渲染、音频播放、口型同步和表情切换。
- 它的口型同步是基于音频音量的实时口型同步，这比我们当前“按音素写死嘴型”更适合短期真实音频驱动。
- 其系统要求是 Python 3.9+、Node.js 18+、AstrBot 3.0+。我们不会引入 AstrBot，但 WebSocket 和音频驱动口型思路有价值。
- 许可证是 MIT，适合参考架构。

### 当前不适合直接复制的原因

- 它是 AstrBot 插件，和我们的比赛项目主线不是同一个运行框架。
- 它依赖 Electron 桌面客户端；Firefly 最终展示更可能是本机全屏显示，不一定需要完整桌面客户端。
- 它的 TTS 集成依赖 AstrBot，不能直接满足我们“端侧/自训练模型”的要求。

### 我们的落地策略

口型同步短期走两层：

1. 有音素/对齐信息时，优先使用音素到嘴型的映射。
2. 没有音素/对齐信息时，使用音频音量包络驱动 `ParamMouthOpenY`。

对真实 TTS 来说，第二层更重要。实现上可以先对 WAV 做短窗 RMS：

```text
音频样本 -> 分帧 -> RMS 音量 -> 平滑 -> 归一化 -> mouth_open 时间线
```

这个方案不理解具体发音，但稳定、简单、能跟真实音频同步，适合作为第一版真实口型。

## 对当前项目路线的调整

我们下一步不再继续加强浏览器纯音，因为它只能证明控件工作，无法代表真实语音。后续路线调整为：

1. 先补运行时事件协议：把 ASR、LLM、TTS、Live2D 控制统一成可传递的事件。
2. 增加真实音频口型链路：从 WAV 或 TTS 输出生成音量包络，驱动嘴巴开合。
3. 增加 TTS 抽象层：主路线改为 VoxCPM2 自托管服务，公网 Space 只用于临时 API 验证。
4. 增加 LLM 到 Live2D 的结构化控制协议：先用规则/假 LLM 生成 JSON，再接本地 LLM。
5. 最后再评估真实 VoxCPM 部署：训练/微调在独立环境完成，板端只承担播放、控制和必要的轻量推理。
